Why we need Git?

Imagine a team of developers working on the same code. Without a system, they might overwrite each other, leading to conflicting versions, i.e. chaos. A Version Control System (VCS), like Git, tracks every single change made to the codebase. This allows developers to work in parallel (via branches), and provides a clear, authoritative history of the entire codebase.

Git solves three core problems:

  • Prevents Loss of Work: Every saved change (a "commit") is a permanent snapshot you can always return to.
  • Enables Parallel Development: You can work on a risky new feature in a branch without messing up the main, working version of the code.
  • Provides Context: The history of commits tells a story of **why** the code is the way it is, which is crucial for maintenance and debugging.

Git Fundamentals

To truly understand Git, you must grasp its three-stage process:

Repository (.git)
  1. Working Directory: Project folder where you actively edit files. Files here are untracked by Git until you tell it otherwise.
  2. Staging Area: Use the git add <file> command to move specific, completed changes from your working directory to the staging area. This allows you to craft a logical, focused "save" rather than just saving everything you've touched.
  3. Repository: This is the permanent, safe storage. When you run git commit, Git takes everything in your staging area and creates a permanent, timestamped snapshot (a commit) in the hidden .git directory.
Essential Git Commands
  • git init: Initializes a new Git repository in the current folder.
  • git clone <url>: Creates a local copy of a remote repository.
  • git checkout <branch>: Switches to a different branch.
  • git switch <branch>: An alternative to checkout for switching branches (simpler syntax).
  • git branch: Lists all local branches and highlights the current branch.
  • git branch -r: Lists remote branches.
  • git fetch: Downloads commits, files, and refs from a remote repository into your local repo, but doesn't merge them.
  • git merge <branch>: Combines the history of the specified branch into the current branch.
  • git pull: Fetches changes from the remote repository and merges them into your current branch.
  • git push: Sends your committed changes to a remote repository.
  • git add <file>: Stages a file for the next commit.
  • git commit -m "<message>": Saves staged changes to the repository with a descriptive message.
  • git status: Shows the status of changes as untracked, modified, or staged.
  • git log: Shows the commit history for the current branch.
  • git diff: Shows the differences between your working directory and the staging area.
  • git reset --hard <commit>: Resets your branch to a specific commit (‚ö† irreversible if commits are lost).
  • git revert <commit>: Creates a new commit that undoes the changes of a previous commit (safe for shared branches).
  • git stash: Temporarily saves uncommitted changes so you can work on something else.
  • git remote -v: Shows the URLs of the remotes linked to your repo.
Typical Git Workflow
# 1. Clone the repository (first time only)
git clone git@github.com:your-org/your-repo.git

# 2. Create and switch to a new feature branch
git checkout -b feat/login-system

# 3. Check the status of your working directory

# 4. Stage changes for commit

# or git add specific-file.js

# 5. Commit changes with a clear message
git commit -m "feat: add login form UI"

# 6. Fetch the latest changes from remote (just in case)
git fetch origin

# 7. Merge or rebase latest changes from dev into your branch
git merge origin/dev
# or: git rebase origin/dev

# 8. Push your branch to remote
git push -u origin feat/login-system

# 9. Open a Pull Request in your Git hosting platform
# (GitHub, GitLab, Bitbucket, etc.)

# 10. After merge, clean up local branch
git checkout dev
git pull origin dev
git branch -d feat/login-system

Git Best Practices for Teams

Branching Strategy

In a team, branches are essential for managing complexity and preventing chaos. A common and effective strategy is GitFlow, which uses different types of branches for different purposes:

  • main: Represents the official, production-ready release history. It should always be stable and deployable.
  • dev: The integration branch for active development. All new features and fixes are merged here first.
  • stag: The staging branch for final testing before production deployment.
  • prod: The branch that reflects the live production environment.
  • feat/*: Feature branches for developing new functionality (e.g., feat/login-system).
  • infra/*: Infrastructure-related changes (e.g., Terraform scripts, deployment configurations).
  • fix/*: Bug fix branches for addressing specific issues (e.g., fix/memory-leak).
  • chore/*: Routine maintenance tasks like dependency upgrades, code formatting, or refactoring (e.g., chore/update-libs).
  • cicd/*: Continuous Integration / Continuous Deployment pipeline-related changes (e.g., cicd/github-actions).

This strategy ensures that the main branch is always clean, and different types of work (new features, infrastructure updates, bug fixes, maintenance tasks, and CI/CD changes) are properly isolated and easier to manage.

Clear and Conventional Commits

A commit is the atomic unit of work in Git. A project with a clean, well-documented commit history is a joy to work on. The industry standard is Conventional Commits, which enforces a simple rule structure for commit messages.

  • feat: A new feature
  • fix: A bug fix
  • docs: Documentation only changes
  • style: Changes that do not affect the meaning of the code (white-space, formatting, missing semi-colons, etc)
  • refactor: A code change that neither fixes a bug or adds a feature
  • perf: A code change that improves performance
  • test: Adding missing tests
  • chore: Changes to the build process or auxiliary tools and libraries such as documentation generation
  • cicd: Changes to pipeline
Pull Requests (PRs)

A Pull Request (or Merge Request in GitLab) is a formal proposal to merge changes from one branch into another. It's the cornerstone of team collaboration.

  • Purpose: It's not just about merging code. It's a forum for code review, discussion, and automated checks (CI).
  • Best Practices for a Good PR:
    • Keep it small and focused. A PR should do one thing well. A 50-line PR is much easier to review than a 5000-line one.
    • Write a clear description. Explain *what* the PR does and *why*. Link to the relevant issue or ticket.
    • Ensure CI passes. Never merge a PR with failing tests.
Tagging Strategy

A tag is a pointer to a specific commit, used to mark important points in history, like releases. The most common strategy is Semantic Versioning (SemVer).

v1.2.3  (Major.Minor.Patch)
  • MAJOR version is incremented for incompatible API changes.
  • MINOR version is incremented for new, backward-compatible functionality.
  • PATCH version is incremented for backward-compatible bug fixes.

This gives clear meaning to your release versions and is what we will use to trigger our deployment workflow.

The "Why": Speed and Stability

In modern software, the biggest challenge is delivering new features to users quickly without breaking things. CI/CD is the cultural and technical foundation that makes this possible. It's an automated assembly line for your code.

Use Case: Imagine a team of 10 developers working on an e-commerce site. Without CI/CD, releases might happen once a month. They are large, risky, and require a "code freeze" and all-hands-on-deck manual testing. With CI/CD, every small change can be tested and deployed independently, potentially dozens of times a day. This reduces risk and gets value to customers faster.

Continuous Integration (CI): The Quality Control Station

CI is the practice of automating the integration of code changes from multiple contributors into a single software project. Its primary goal is to provide a rapid feedback loop.

Best Practice: The Build is Sacred. If the `main` branch build fails, it becomes the team's number one priority to fix it. A broken `main` blocks everyone and defeats the purpose of CI.

Continuous Deployment/Delivery (CD): The Shipping Department

CD automates the release of validated code to a production environment. It's crucial to distinguish between two types:

  • Continuous Delivery: The CI pipeline verifies the code and packages it for release. The final deployment to production is triggered by a manual button press. This allows for business-level sign-off.
  • Continuous Deployment: This is the ultimate goal. If the CI pipeline passes, the code is automatically deployed to production without any human intervention. This requires a high degree of confidence in your test suite.

Best Practice: Pipeline as Code

Your CI/CD pipeline is a critical piece of infrastructure. It should be treated like code. By defining your pipeline in a YAML file (like deploy.yml) that lives inside your repository, you gain several advantages:

  • Version Control: Your pipeline's history is tracked in Git, just like your application code. You can see who changed it and why.
  • Reproducibility: Anyone on the team can check out the repository and understand (and potentially run) the entire build and deployment process.
  • Collaboration: Changes to the pipeline can be reviewed and approved via Pull Requests, just like any other code change.

This principle, often called GitOps, is a cornerstone of modern DevOps practices. It makes your entire software delivery process more transparent, reliable, and collaborative.

GitHub Actions

GitHub Actions is a CI/CD platform built directly into GitHub. It allows you to automate your software workflows right from your repository. You can build, test, and deploy your code without needing an external service.

Key Components:

  • Workflow: An automated process defined by a YAML file (like our deploy.yml). A repository can have multiple workflows.
  • Event: A specific activity that triggers a workflow. Examples include a push to a branch, a pull_request being opened, or a tag being created.
  • Job: A set of steps that execute on the same runner. Jobs run in parallel by default.
  • Step: An individual task that can run commands or an "action."
  • Action: A reusable piece of code that performs a complex but common task. For example, actions/checkout@v4 checks out your repository's code onto the runner.
  • Runner: A server (usually a virtual machine) that runs your workflows. GitHub provides runners, or you can host your own.

Secrets vs. Variables: In our lab, we use both. Secrets are for sensitive information like your AWS_SECRET_ACCESS_KEY. They are encrypted and are not visible in the logs. Variables are for non-sensitive configuration data like your AWS region or S3 bucket name. They are visible in the logs.

Common Workflow Examples

Running Tests on Pull Requests: This is a fundamental CI practice. This workflow triggers on any push to a pull request, checks out the code, sets up the environment (e.g., Node.js), and runs the test suite.


name: Run Tests

on:
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Install dependencies
        run: npm install
      - name: Run tests
        run: npm test

Deploying on Tag Push: This is the workflow we use in our lab. It ensures that only specific, versioned commits are deployed to production.

  1. Create IAM User: In AWS IAM, go to Users > Create user. Name it `github-actions-user`. On the permissions screen, select "Attach policies directly" and add the `AmazonS3FullAccess` policy. Create the user.
  2. Generate Access Keys: Click on the new user, go to the "Security credentials" tab, and click "Create access key". Choose "Command Line Interface (CLI)", confirm, and create the key. Save the Access Key ID and Secret Access Key.

  4. Enable Public Access: In the "Permissions" tab of your bucket, turn OFF "Block all public access" and save the changes.
  5. Add Bucket Policy: Still in the "Permissions" tab, edit the Bucket policy and paste the following JSON, replacing `YOUR_BUCKET_NAME` with your actual bucket name.

    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "PublicReadGetObject",
            "Effect": "Allow",
            "Principal": "*",
            "Action": "s3:GetObject",
            "Resource": "arn:aws:s3:::YOUR_BUCKET_NAME/*"
        }
    ]
}
  6. Enable Static Hosting: In the "Properties" tab, enable "Static website hosting" and set the index document to index.html. Note the website endpoint URL.

  1. Make Bucket Private: In your S3 bucket's "Permissions" tab, turn ON "Block all public access" and delete the public bucket policy.

    {

    name: Deploy to AWS S3 and CloudFront

on:
  push:
